1. 何を作ったか

Gmail と連携し、受信したメールを自動で詐欺かどうか判定し、「なぜ危ないか」を非ITの人にも分かる日本語で説明するアプリです。受信メールの自動判定が本命ですが、誰でもログイン不要で、怪しいメッセージを貼り付けて今すぐ試すこともできます。

※ 公開URLでは、ログイン不要の貼り付け／サンプルで誰でも判定を体験できます。Gmail 連携（自動判定）はテストユーザーで動作検証済みで、全ユーザーへの一般開放は Google の OAuth 公開審査中です。

判定して終わりではありません。裏側では攻撃エージェント（狼）が検知器の死角を突く型を進化させ、防御エージェント（番犬）がそれを学習して捕まえ返す攻防が回り、検知器が自分で強くなり続けます。

2. なぜ作ったか

狙われているのは、中小企業の非IT層——警察から詐欺注意の警告を受けるような経営層です。彼らに刺さるのは、生成AIで自然になった日本語の巧妙な詐欺で、ルールベースのフィルターは引っかけられません。しかも既存フィルターは「ブロックする」だけで理由を説明しないので、利用者は次も同じ手口に引っかかります。

攻撃が AI で進化するなら、防御も進化し続ける仕組みでなければ追いつけない。そこで検知を一度きりのモデルではなく、自己改善し続けるパイプラインとして設計しました。

DevOps × AI Agent への翻訳

攻撃⇄防御の攻防は、検知器の CI/CD パイプラインそのものです。

攻防の動きDevOps の工程

すり抜けた型を事例DBへ書き戻しパッチ
次ラウンドで再判定継続的デリバリー
攻撃が型を組み替えて再テスト回帰テスト

この CI/CD をAI エージェントが人間の介入なしに回すのが KangaL の核です。ひとつの自己改善ループが「つくる」と「まわす」の両方を満たします。

3. 安全設計の背骨 — 完成した詐欺文をつくらせない

攻撃エージェントにそのまま送れる完成詐欺文を作らせません。詐欺を「完成文」ではなく 6つの心理レバーの設定値の集合（AttackPattern）として表現します。攻撃側はこのレバーの組み合わせ（型）を進化させ、防御側はメッセージから同じレバーを逆算します。完成詐欺文はどこにも生成・保存されないので、悪用可能な危険物が残りません。

6レバー：緊急性 / 権威（なりすまし）/ 報酬・恐怖 / 誘導 / 個人化 / 孤立化。うち「個人化」と「孤立化（内密に・相談するな）」が BEC（ビジネスメール詐欺）を捉える差別化レバーです。

4. どう動くか（防御エージェント＝主役）

  1. 構造分解 — Gemini がメッセージから6レバーを逆算。
  2. 能動調査 — anchor の matchKnownScams（事例DB照合）を必ず1回。加えて URL レピュテーション（Web Risk）／ドメイン年齢（RDAP）／送信者認証（SPF・DKIM・DMARC）／公的アラート照合の4ツールを、入力の特徴を読んでモデルが動的に選ぶ（function-calling）。
  3. 総合判断 — レバー×重みの素点に調査の危険シグナルを加点のみで重ね、確信度スコアと「なぜ危ないか」の日本語説明を生成。
  4. 蓄積 — すり抜けた型を Firestore に書き戻し、次の照合材料にする（自己改善）。

取引先になりすました BEC（「お振込先変更（社外秘）」＝孤立化レバー）を確信度90で赤と判定し、理由を非IT向けの日本語で説明します（画像欄参照）。確信度はあくまで UI 上の安心度表示で、検知性能の主張ではありません（性能は §6 の recall / FPR / holdout を参照）。

5. 技術スタック（Google 純正）

Next.js (TypeScript) ／ @google/genai（Vertex 経由・Gemini 2.5 Flash）／ Cloud Run ／ Firestore ／ Google Web Risk（本番配線済み）／ RDAP。

Cloud Run へ鍵レス（ADC）でデプロイ。サービスアカウント JSON をリポジトリにもイメージにも置かず、実行 SA の身元で Vertex を叩く＝シークレットの正しい扱いです。

協調ループは手組み（正直に）

攻撃⇄防御の協調は loop.ts の手組みオーケストレーションで実装。唯一の動的判断＝調査のツール選択は既に function-calling にあり、ここにマルチエージェントフレームワークを被せても利得が薄いという判断です。フレームワークに頼らず協調を組んだ、というのが実装の事実です。

6. 測定の誠実さ（ここが KangaL の主張）

「賢く育っている」を雰囲気で言いません。検知力は単発スコアではなく recall / FPR を主軸に測ります。recall 単体は「全部クロ」と言えば100%になる罠で、非IT向けは誤検知（オオカミ少年化）こそ致命的なので、誤検知を増やさず recall を育てることを指標にしました。

  • 閉ループの学習：すり抜けた型を書き戻すと、次ラウンドで照合が捕捉する。デモの「検知率が上がるアーク」はこの機構。
  • 未見型への汎化：照合器ブラインドで作った未見型に対し、能動骨格（権威＋個人化＋孤立化）が一致すれば誘導手段が違っても捕捉（robust 2件・系統非依存まで確認）。
  • 実物サンプルへの実力（外部 holdout・n=6）：フィッシング対策協議会＋実受信の本物詐欺を物理隔離して評価。recall は判定床に依存し、床70で 1/6・床60で 4/6 [30–90%]（Wilson 95% CI）。小標本のため最適な床は点で確定できずバンドとして提示。陽性所見として missed-perception=0＝「攻撃と知覚はできている。厳しいのは判定床のほう」。

隠さない限界：商用良性サンプル（誤検知の本番ストレッサー）は評価途上＝床を下げたときの真の誤検知リスクはまだ測定中。subtle な非executive BEC（語調を和らげた送金・認証情報要求型）は床落ちで取りこぼします。原因は intensity に下限が無く判定にも一般床が無いこと。これは自己改善ループ自身が炙り出した現在地で、行動レバーに床を設ける方向で誤検知測定とセットで校正中です（床の最終決定は人間ゲート）。

この「できること・できないことを数字の解像度で言い分ける」姿勢自体を、評価軸（取り組みの一貫性・実用性）への回答として提出します。

7. デモの位置づけ（盛らない）

/demo の攻防アークは実走の決定論リプレイ（録画用に固定したシナリオ）であり、その場でライブの攻撃AIが回っているわけではありません。協調が本物であることは / のライブ判定（実 Gemini パイプライン）が担保します。デモ動線では攻撃エージェントは HTTP 表面に露出しません（モードゲート＋ルート分離）。「ライブ協調」と「録画デモ」を混同させないことが、この提出物の信頼の土台です。
