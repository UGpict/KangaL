## 0. このメモの目的

設計ドキュメント v0.5（§8 で「貼り付け→判定」を通常モードの主役、§10 で Gmail/Teams 監視を「やらない・将来構想スライド1枚」に降格）に対し、
**本当に作りたいのは「メール/メッセージの自動検知」だった**という前提のズレを反映する。
「何を・なぜ・どのスコープで」の意思決定記録。

## 1. 結論（先に）

| 論点 | 結論 |
|---|---|
| 自動検知はやるべきか | **やるべき寄り。** やらないと評価軸「とどける（実運用）」が弱くなる |
| 貼り付けUI | **降格 or 廃止。** 守りたい層（怪しいと気づかない非IT層）に最も届かない形 |
| Gmail全機能実装 | **不要。** 「軽い実装（読み取りポーリング）」で本物を一通通せば十分 |
| 攻防ループ | **据え置き・主役のまま。** 評価軸「必然性・自律性」にドンピシャ |

**三つの動詞への割り当て:**
- つくる/まわす = 攻防ループ（シミュレーションで作り込む）
- とどける = 本物 Gmail を一通

## 2. なぜ「貼り付け」を降格するか（プロダクト論理）

TascaLL の致命傷だった「いつ使うんだ問題（習慣の壁）」が、貼り付けUIにも同じ穴として存在する。
怪しいと思った人しか貼らない。だが詐欺に一番刺さるのは**「怪しいと思わなかった人」**。
守りたい相手（警察に注意される社長層）は、わざわざ怪しいメールをコピペして貼らない。気づいてないから刺さっている。

→ 貼り付けは「守りたい相手に最も届かない形」。自動検知が正しい。受信箱に座り、本人が気づく前に番犬が吠える。

プロダクト名（番犬 Kangal）とも整合：
- 貼り付け = 飼い主が「これ嗅いで」と差し出す犬
- 自動検知 = 勝手に見回る番犬 ← **欲しいのはこっち**

## 3. ハッカソン要件の確認結果（2026/06/05 時点・一次ソース）

参照：Google Cloud 公式ブログ（2026/05/08）／ Findy プレスリリース（2026/04/27）／ connpass イベントページ スニペット

**効いた事実 3 つ:**

1. **「とどける（本番品質・実運用）」が最重要級。**
   - 公式：「動くものをつくる、ではなく、届くものをつくる。そこまでをゴールに置いている」
   - Findy：「プロトタイプは作れるが、実運用まで持っていけない」という課題に向き合うのが趣旨。
   - → デモ用受信トレイのシミュレーションだけだと、この軸を正面から外すリスク。

2. **最重要評価軸は「AIエージェントである必然性」。**
   - connpass：「AIエージェントが価値の中心になっているか：自律的な振る舞いがあるか、"AIエージェントである必然性"があるか」
   - → KangaL の攻防ループ（防御エージェントが入力を読んで調査ツールを動的選択、攻撃⇄防御が自律で回る）がドンピシャ。

3. **必須技術と任意技術の切り分け。**
   - 必須：Gemini Enterprise Agent Platform / Gemini / ADK / Cloud Run（Bootcamp で扱う技術群）。
   - 任意：「Flutter、Firebase、Veo、Elasticsearch ほか、任意の技術を利用可能」。
   - → **Gmail API 連携は「任意の技術」扱い。自由に足せる。** 必須技術構成（ADK+Gemini+Vertex+Cloud Run）は v0.5 のままでクリア。

> 補足：v0.5 は「ADK (TypeScript)」を採用。公式必須技術名は「Gemini Enterprise Agent Platform」も併記。Bootcamp 資料で ADK と Gemini Enterprise Agent Platform の関係（ADK がその上で動く想定か）を一度確認しておくと安全。

## 4. 三つの動詞への役割分担（設計の core）

| 動詞 | 対応する評価軸 | KangaL での実装 | データ |
|---|---|---|---|
| つくる / まわす | 必然性・自律性（最重要） | 攻防ループ＝検知器の自己改善 CI/CD。ここに一番時間をかける | シミュレーションでOK（閉ループ内） |
| とどける | 実運用・本番品質 | 「本物のGmailに本物のメールが届いて自動で番犬が吠える」を一点だけ見せる | 本物の Gmail（自分のテストアカウント） |

物語の接続：「自動で立ったフラグの精度が、攻防ループで日々上がっていく」と繋げば一本の筋になる。
→ 「番犬が勝手に見回る、しかも日々賢くなる」。むしろ繋がると今より強い。

## 5. Gmail 連携：軽い実装 / 重い実装の切り分け（5週間の生死を分ける）

### ❌ 重い実装（避ける）
- Gmail push 通知（Pub/Sub watch）でリアルタイム新着検知。
- OAuth 審査・スコープ申請・Pub/Sub トピック設定で時間が溶ける。
- デモで「リアルタイムに勝手に」を見せるためだけにここまでやるのは、5週間ソロだと割に合わない。

### ✅ 軽い実装（おすすめ）
- `messages.list` / `messages.get` を読み取りスコープ（`gmail.readonly`）で自分のテストアカウントに対して叩く。
- デモ当日：「いま受信箱を見に行きます」ボタン → 直近の未読を1〜数通取得 → 防御エージェントに流す → 自動でフラグ。
- push 通知の重さを回避しつつ「本物の Gmail から本物のメールを取って自動判定した」は完全に成立。
- 「リアルタイム監視」は将来構想スライドに置く。本番品質の証明にはポーリングで十分。
- 工数感：認証まわりは `/apa/` 等で経験済みの土俵。数日〜1週間に収まる見込み。

### ⚠️ 先に潰しておく誤解
- Gmail 読み取りでも本番公開アプリ化には Google の OAuth 審査が要る。
- だがハッカソンデモは「**テストユーザーに自分のアカウントを登録した未審査アプリ**」で完全に動く（テストユーザーなら審査不要）。
- デモは自分の画面で自分の Gmail を見せるだけなので、これで困らない。
- **「審査通さなきゃ」と思い込んで1ヶ月構えるのが最大の罠。やらない。**

## 6. 前回からの積み残し懸念（v0.6 で一緒に詰めたい）

> 自動検知とは別軸だが、評価の誠実さ（§7・§11）に直結するので並行で。

### 6-1. 閉ループの過適合（自家中毒）リスク
- §7 末尾「recall の分母は攻撃側が自動供給」→ それで測った recall は「自分が出した型を自分で捕まえた率」でしかない。
- 「検知率が目の前で上がる」のギザギザは閉ループ内の現象。番犬が賢くなったのか狼のクセを覚えただけか、ループ内側からは原理的に区別不能。
- **Google 審査員（ML 分かる人）はここをほぼ確実に突く。**「その recall、攻撃生成器に過適合してるだけでは？」

### 6-2. 解：外部ホールドアウト（§12 の偵察先を評価専用に転用）
- フィッシング対策協議会・IPA・JPCERT-CC・警察庁の公開事例を、**攻撃エージェントには一切食わせず評価専用ホールドアウト**として取り分ける。
- 手口の型は偵察に使うが、評価に使う実物サンプルは攻撃側から隔離する。
- 主張がこう変わる：
  - ループ内 recall（自家生成）= 学習が回ってる証拠
  - **ホールドアウト recall（外部実物・攻撃側未見）= 実際に人を守れる証拠**
- この二本目が動いて初めて「動くデモ」が「本当に人を守るもの」になる。「外部ホールドアウトで検証してます」と先回りで言えること自体が差別化シグナル。
- coverage（§7 補助指標）も同様：自分の攻撃側の組み合わせ空間のカバー率なので、外部ホールドアウトとセットで語らないと内輪の数字に見える。
