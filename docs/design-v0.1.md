## 1. プロダクト概要

### 何を
怪しいメッセージ（メール・SMS・チャット文面）を入力すると詐欺かを判定し、
**「なぜ危ないか」を非ITの人にも分かる日本語で説明**するアプリ。

### 誰に
中小企業の非IT層（例: 警察から詐欺注意の警告を受けるような社長層）。
既存フィルターが拾えない、自然な日本語の巧妙な詐欺が刺さる層。

### なぜ
- 既存フィルターはルールベースで「ブロックする」だけ。理由を説明しない。
- 最近の詐欺は自然な文章で来るため、ルールに引っかからない（生成AIで文面が自然化＝検知の難所）。
- 攻撃が進化し続けるから、防御も進化し続ける仕組みが要る。
- 「ブロック」ではなく「なぜ危ないか分かって、次は自分で気づけるようになる」教育効果を狙う。

## 2. コアコンセプト（勝ち筋）

**攻撃エージェント（狼）と防御エージェント（番犬）が自律的に攻防し、検知精度を育て続ける。**

DevOpsの言葉に翻訳すると、検知器のCI/CDパイプラインをAIエージェントが自律的に回している状態:

```
攻撃が検知器の死角を発見   = 脆弱性スキャン
　↓ 防御ルール/プロンプト自動更新  = パッチ生成
　↓ 更新版で再判定               = 継続的デリバリー
　↓ 攻撃が再テスト               = 回帰テスト
　↓ 検知率の推移を可視化          = 監視
```

→ **DevOps × AI Agent の両テーマを同時に射抜く。**

---

## 3. 設計の大原則 ―「道B」

攻撃エージェントには **実弾（そのまま送れる完成された詐欺文）を作らせない。**
代わりに **攻撃の型・戦略**（＝ `AttackPattern`）を進化させ、検知器の死角を探す。

| | 道A（不採用・危険） | 道B（採用） |
|---|---|---|
| 攻撃側の出力 | 完成された詐欺文を生成・蓄積 | 攻撃の型・戦略を進化させる |
| 理由 | デュアルユースの危険物が残る | 防御を鍛えるのに実弾は不要 |
| 規約 | Google生成AIポリシー違反リスク | セーフ |

## 5. 中核データモデル: `AttackPattern`【確定】

このシステムの**背骨**。詐欺を「完成文」ではなく「心理レバーの設定値の集合」として持つ。
完成した詐欺文を一切持たない（＝道Bの実装そのもの）。

**1つの`AttackPattern`が3つの役割を同時に支える:**
1. 攻撃が**書く**（レバーを組み立て・変異＝型の進化）
2. 防御が**読む**（入力から各レバーを逆算＝構造分解）
3. **危険度スコアの素**（各レバーの `intensity` × 重み）

```typescript
type AttackPattern = {
  id: string;
  generation: number;        // 何世代目の進化か
  sourceContext: string;     // 偵察の起点（例: "フィッシングレポート2026 / CEO詐欺"）
  channel: "email" | "sms" | "line" | "voice_deepfake" | "web";  // メタ：媒体

  levers: {
    urgency: {                                  // ① 緊急性
      tactic: "deadline" | "account_freeze" | "limited_offer" | "none";
      intensity: 0 | 1 | 2 | 3;
    };
    authority: {                                // ② 権威
      impersonates: "financial" | "government" | "business_partner"
                  | "executive" | "delivery" | "platform" | "none";
      credibilityTricks: ("logo_mimicry" | "formal_tone"
                        | "reference_number" | "url_lookalike")[];
    };
    incentive: {                                // ③ 報酬/恐怖
      type: "reward" | "fear";
      hook: "prize" | "refund" | "penalty" | "account_loss" | "legal_threat";
      intensity: 0 | 1 | 2 | 3;
    };
    callToAction: {                             // ④ 誘導
      action: "click_link" | "transfer_money" | "input_credentials"
            | "call_number" | "install_app" | "scan_qr";
      friction: "low" | "mid" | "high";
    };
    personalization: {                          // ⑤ 個人化（差別化の核心）
      level: "broadcast" | "segmented" | "targeted";
      signals: ("real_name" | "transaction_history"
              | "thread_injection" | "internal_jargon")[];
    };
    isolation: {                                // ⑥ 孤立化
      tactic: "secrecy" | "bypass_approval" | "direct_channel" | "none";
      intensity: 0 | 1 | 2 | 3;
    };
  };

  detectionResult?: { detected: boolean; missedBy?: string; };
};
```

### 設計判断の注記
- **⑤ personalization が差別化の核心。** `thread_injection` が BEC の「いつものやり取りに割り込む」を表現。
- **⑥ isolation を独立レバーに。** 「内密に・相談するな」は正規依頼ではまず出ない → 単独で強い赤信号。
- **channel はメタ情報。** 心理操作（レバー）と配送経路（channel）を分離。

## 6. エージェント構成

### 6-1. 攻撃エージェント（狼）― 進化し続ける脅威源
偵察（公的機関の公開情報を収集）→ 戦略立案（防御の死角を突くレバー組み替え）→ 出力（`AttackPattern`）。

### 6-2. 防御エージェント（番犬）― 主役
```
① 構造分解   : メッセージから AttackPattern の各レバーを逆算
② 能動調査   : ← 主役。ツール群を「入力に応じて」呼ぶ（§6-3）
③ 総合判断   : 危険度スコア（レバー×重み）＋「なぜ危ないか」日本語生成
④ 蓄積       : 判定を事例DBに保存（次の照合材料 ＝ 自己改善）
⑤ フィードバック : 攻撃側に detectionResult を返す
```

### 6-3. 調査ツール群【確定】

| ツール | 手段 | 返すもの | 効くレバー |
|---|---|---|---|
| `checkUrlReputation` | Web Risk Lookup API | 安全/危険＋脅威種別 | ②④ |
| `checkDomainAge` | RDAP | 登録日・経過日数 | ②（登録直後＝なりすまし疑い） |
| `verifySenderAuth` | SPF/DKIM/DMARC 自前パース | 認証pass/fail | ②⑤ |
| `matchKnownScams` | Firestore事例DB照合 | 類似事例＋類似度 | 全体 |
| `checkOfficialAlerts` | フィッシング対策協議会の緊急情報照合 | 対象が最近報告されているか | ②+sourceContext |

各ツールの `description` に「いつ使うべきか」を書く＝エージェントの呼び出し判断材料。

### 6-4. オーケストレーション【確定】― 動的ルーティング寄りハイブリッド
大枠（構造分解→調査→判断）はワークフローで固定。調査フェーズ内のツール選択はエージェントに委ねる。
「読んだ結果から次の一手を選ぶ」自律性が単発推論との差でありデモの見せ場。全ツール順次実行は不採用。

## 7. 検知メトリクス【確定 / v0.4】― デモの心臓

「検知率が目の前で上がる」を成立させる核。**何を測るかでデモの説得力が決まる。**

### 主軸（2軸で見る）
- **① 検知率（recall）**: 攻撃エージェントの型をどれだけ捕まえたか。
- **② 誤検知率（FPR / precisionの裏）**: 正規メッセージをどれだけ誤って詐欺と判定したか。

**recall単体は罠。** 「全部クロ」と言えば100%になるが誤検知だらけで実用にならない。
特に非IT向けは**正規メールを詐欺と誤判定する方が致命的**（オオカミ少年化）。
→ 2軸で見ることで、デモのストーリーが強くなる:
攻撃が新しい型を出す → 検知率が一旦下がる → 防御が学習 → **誤検知を増やさず検知率が戻る**。
この「攻防のギザギザ」＋「誤検知の低位安定」が同時に見えると「賢く育ってる」が伝わる。

### 補助指標（KangaL独自）
- **③ カバーした型の数（coverage）**: 6レバーの組み合わせ空間のうち対応できた型の増加。
  「`personalization=targeted` × `isolation=secrecy`」のような手強い組み合わせを攻略した、と型ベースで言える。
  `AttackPattern` を背骨にした設計だからこそ出せる、他にない見せ方。

### 審査員向けフレーミング（二枚舌）
- **UI（非IT向け）**: 優しい日本語（「危険度」「なぜ危ないか」）。
- **口頭説明（Google審査員向け）**: ML語彙に翻訳して語る。
  検知率＝recall、誤検知の低さ＝precision、両立＝F1、攻撃進化での精度低下と回復＝敵対的robustness。
  → 「敵対的サンプルに対するrobustnessを、recall/precisionのトレードオフを保ったまま上げている」。
- **先回りが効く**: 「recallだけ上げるのは簡単（全部クロと言えばいい）。だが実用にならないので
  誤検知率を抑えたままrecallを育てることを指標にした」と**聞かれる前に言う**＝「分かってる人が作ってる」信号。

### 評価の土台（重要な前提）
誤検知率(FPR)の分母は**正規メッセージ**。これが無いとFPRが測れない。
攻撃エージェントは良性サンプルを作れない（道B的にも作らせたくない）。
→ **良性サンプルセットを別途仕込む**必要がある（§13 未決）。
recallの分母（詐欺サンプル）は攻撃側が自動供給するが、FPRの分母は人間が用意する。

## 9. 安全設計（道Bに合わせて仕分け済み）

### 入れる
- 架空の固有名詞（実在名は使わずカテゴリのみ）/ ラウンド数ハード制限（5〜10）/
  データソースを公的機関の公開情報に限定 / モード分離 / 倫理声明（README＋アプリ内＋デモ冒頭末尾）/
  **Prompt Injection対策**（外部取得テキストは「命令」でなく「データ」扱い。`checkOfficialAlerts`で特に効く）。

### 入れない（道Bでは危険物が生まれないため対象が存在しない）
- 透かし強制 / 暗号化保存 / 人間レビューゲート / 出力フィルタ事後ブロック / OWASP LLM Top10フルセット
